## プロジェクト概要

本プロジェクトは、ブラウザ環境で完全に完結する、エンジニアやパワーユーザー向けの高速・高機能なPDF抽出・分割Webアプリケーションです。

既存の無料オンラインPDFツールの多くは、ページを一つずつマウスクリックで選択するビジュアル重視のGUIを採用しています。これらは直感的である反面、大量のページ処理やキーボード主体のワークフローには不向きです。さらに、学術論文や技術書などで多用される「論理ページラベル（ローマ数字など）」での範囲指定に対応したクライアントサイドツールは事実上存在しません。

本アプリケーションは、CLIツールのような「テキストベースでの柔軟かつ高速なページ指定」という体験をWebブラウザ上に持ち込むことを目的としています。サーバーサイドの処理を一切持たず、クライアントサイド（JavaScript）のみでPDFの解析と再構築を行うため、高い機密性を保ちながら、ゼロコストでの運用（GitHub Pages等での静的ホスティング）を実現します。

コアバリューと主要機能
コマンドライクな操作性 (String-based Pagination): 1,3,5-7 のようなテキストによる柔軟なインデックス指定に対応し、マウス操作を最小限に抑えた直感的で高速なページ抽出を実現します。

論理ページラベルの解釈 (Logical Label Parsing): PDFのメタデータからページラベル（i, ii, iii... 等）を自動解析し、i-v のような人間にとって自然な範囲指定を可能にします。これは目次と本文でページ番号の体系が異なるドキュメントを扱う際に極めて強力に機能します。

プライバシーファースト (Zero-Server-Transmission): ユーザーのPDFファイルは外部サーバーに一切送信されません。pdf-lib および PDF.js を用いてすべての処理をブラウザのローカルメモリ上で完結させるため、機密性の高い文書も安全に扱うことができます。

ゼロ・メンテナンス (Static Hosting Ready): バックエンドAPIやデータベースへの依存が一切ないため、GitHub Pagesなどの静的ファイルホスティングサービスにデプロイするだけで、半永久的かつ無料で稼働し続けます。

## 開発ルール

### 積極的にコマンドを実行すること

ただし以下のルールを守ること

- Nodejs系コマンドは全て`pnpm`を介して行うべきであり、`wrangler`, `prisma`, `biome`などの直接（グローバルインストールを期待した）実行や`npx`を勝手に実行することは禁止
- 実験などのために`node`を実行するのは許可される。
- `pnpm dev`など、long runningなコマンドは勝手に実行しないこと
- .env等のgitignoreされた機密情報の読み取り禁止
- 開発環境のシステムの状態に影響を及ぼす可能性のあるコマンドが必要な場合は、実行前に私に許可を求めること

### 書き込み後の品質担保

あなたがまとまった変更をしたあと、以下のコマンドを実行すること
- `pnpm run typecheck`
- `pnpm run fix` (biomeの実行)

もしこれでエラーが起きるのなら、止む終えない場合を除き、適宜修正することが望ましい。
したがって原則として、**エラーを残したまま次のまとまった作業に進むことは禁止する**

### ドキュメントでの番号振りは禁止

ドキュメント内で箇条書きにや見出しなどに番号を振ることは禁止する。
ドキュメントが更新されるたびに番号がずれてしまい、メンテナンスコストが高くなるためである。

禁止例
```markdown
## 1. はじめに // "1.", "a."のような振り分け禁止
## 2. インストール
## 3. 使い方
```

### 先験的にすべてを予測できない

はじめは容易に実現できると思っていたことや、やるべきだと思っていたことが、よくコードを読んでみると実は難しかったり、そもそもやるべきではなかったりすることがある。もしそう感じたら、作業を中断して必ず私に確認を取ること。

### いいコードは、コメントを必要としない

コードをコメントで補ったり解説するのではなく、それが不要であるほど可読性の高いコードを追求すること。

コメントは、「スキルの高いエンジニアが見ても自明ではないこと」のみに使う手段であり、例えば以下のケースで許される：

- ソースコード外部の特殊な事情によりコードも特殊になっている場合の注釈
- DocStringなどの記述が期待される共通関数・ライブラリなど
- `// TODO`などの一時的マーク
- Userが学習過程である場合の教育的サポート

### The best writing is rewriting

コードはいつだって不完全。良いコードとは、リファクタによってのみ生まれる。

あなたはAIである性質上、とにかく正常系の動作クリアをするまでコードを生成するのに長けるあまり、リファクタを怠って技術負債を溜め込みがちである。積極的に自らリファクタTaskを挿入すること。
例：「このクラスはSRPに違反しているので分割します」「一時的な`any`アサーションを削除して正確なTypingを試みます」など。

「確定版」や「最終」「final」などの表現を使うことは禁止する。コードもドキュメントも常に改善の余地があるからである。
